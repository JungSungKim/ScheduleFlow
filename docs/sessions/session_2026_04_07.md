# ScheduleFlow 세션 작업 내역 및 회고 (2026.04.07)

## 1. 주요 구현 완료 사항
1. **대시보드 레이아웃 개편 및 반응형 적용**
   - 화면 사이즈에 구애받지 않고 가독성을 유지하기 위해 카드 최소폭(320px) 보장 및 단 바꿈(Wrap) 방식 도입.
   - 카드가 무한정 길어지는 것을 방지하기 위해 내용이 많은 카드의 내부 스크롤 높이(max-height: 280px) 제한.
   - 모바일 환경 화면 폭(768px 이하)에서 캘린더 셀 크기, 폰트 비율, 패딩 등을 최적화하는 미디어 쿼리(`@media`) 적용.

2. **사용자 임의 배치 (Drag & Drop) 기능 구현**
   - 카드 상단에 생긴 핸들바를 터치하여 대시보드의 카드 순서를 위아래로 끌어다 바꿀 수 있는 기능 추가.
   - 변경된 배열은 브라우저 로컬 스토리지에 저장되어 앱을 껐다가 켜도 사용자 설정이 유지되도록 구현.

3. **대시보드 미니 캘린더 업그레이드**
   - 메인 캘린더에서 완성된 바(Bar) 모양의 연속성 띄우기와 겹침(Stacking) 알고리즘을 대시보드 미니 캘린더에도 동일하게 적용.
   - 대시보드에서는 최대 5개의 일정을 펼치고, 이후 초과분은 `+N개 더보기`로 깔끔하게 정리.
   - 달력 내의 일정 하나를 클릭/터치할 때 바로 해당 상세 모달(수정 창)이 뜨도록 이벤트 연동.

4. **단축 홈(대시보드 복귀) 버튼 배치**
   - 어느 화면에 들어가 있든 즉시 대시보드로 되돌아갈 수 있도록 햄버거 메뉴 옆(탑 헤더 좌측)에 전용 홈(Home) 버튼 추가.

## 2. 발생한 실수(번복 내역) 및 해결 전략

### ⚠️ 아이콘 렌더링 누락 및 CSS 간섭 버그
- **문제:** 카드의 위치 변경을 위한 드래그 핸들로 '점자 아이콘(⠿)'을 사용했으나 특정 모바일 브라우저 폰트에 의해 투명 처리되는 렌더링 이슈 발생. 게다가 기존 할 일에서 쓰던 CSS 클래스(`todo-drag-handle`)를 복사해 사용하면서 모바일(Hover 불가 환경)에서 아이콘이 강제 숨김 처리되는 코드 실수를 범함.
- **해결 (번복):** 가장 범용적인 햄버거 아이콘(`☰`)으로 변경하고, 대시보드 전용인 `dash-drag-handle` CSS 클래스를 분리, 독립시켜 상시 표시되도록 정정.

### ⚠️ 모바일 Native Drag & Drop 채터링 (화면 떨림) 현상
- **문제:** 모바일 터치를 대응하기 위해 구형 HTML5 Polyfill을 사용했으나, 모바일 브라우저의 스크롤 판정과 충돌하며 카드가 사방으로 깜빡이는 채터링(Jittering)이 극심하게 발생. 또한 드롭할 예상 위치에 대한 시각적인 피드백(가이드 박스)이 없어 극히 불편하다는 지적을 받음.
- **해결 (전면 개편):** 이 방식이 UX를 망친다고 판단, 즉각 코드를 폐기하고 안정된 업계표준 라이브러리인 **SortableJS** 렌더링 구조로 모듈 교체. 부드러운 애니메이션(200ms)과 반투명 타겟 박스(Ghost Element) 등 앱 완성도를 폭발적으로 끌어올림.

### ⚠️ 홈 버튼 위치 선정 판단 착오
- **문제:** 대시보드로 돌아가기 위한 홈 버튼을 최초에 우측 상단의 `[+ 새 일정]` 옆에 배치함. 그러나 기존 햄버거 메뉴가 좌측 메인이라는 시선 관습을 무시한 설계였고 모바일 환경에서 사용자의 눈에 쉽게 띄지 않는 동선이었음.
- **해결 (재배치):** 사용자 피드백 즉시 위치를 화면 좌측 가장 상단(햄버거 버튼과 현재 페이지 이름 사이)으로 되돌리고, 이모지 문자가 아닌 모든 환경에서 선명하게 보이는 SVG 코드로 교체해 눈에 띄게 강조.

### ⚠️ 브라우저의 끈질긴 캐시(Cache) 문제 대응
- **문제:** 스크립트(.js) 및 프레임워크 CSS 내부의 버전 파라미터(`?v=12` 등)를 여러 번 배포했으나 정작 모바일 브라우저가 진입점 그 자체인 최상단 `index.html` 파일을 강력하게 캐싱하고 있어 화면 업데이트가 과거로 나타남. 
- **교훈:** 파일 내부 자원의 파라미터 갱신만 믿을 것이 아니라, 호스팅 완료 시 사용자에게 **URL 파라미터 조작 주소(예: `?v=new`)를 전달하거나 하드 리프레시를 유도**하여 가장 바깥쪽 캐시를 뚫어내는 커뮤니케이션을 간과하지 않아야 함.
